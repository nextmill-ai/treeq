## CONTEXT.md

The architectural ground-truth doc. ## ROADMAP.md

The product strategy doc, captured 2026-05-09. Frames TreeQ's evolution from "calculator" to "pricing engine": today it prices a single tree's dismantling, but the product vision is to price the entire trade by combining tree + crew + resources + site + post-cut. Lays out a 10-phase plan (P1 favorites/no-auth → P2 auth foundation → P3 saved trees + GPS → P4 resources → P5 team mgmt → P6 Jobber integration → P7 pricing engine v1 → P8 other FSMs → P9 PHCRx → P10 trimming model). Specifies 7 features (F1–F7) including Supabase-vs-Clerk auth comparison (recommends Supabase as primary), curated F4 resource taxonomy (powered-equipment+operator bundles, tonnage/size matrices for cranes/lifts/bucket trucks, materials, deliberately-omitted hand tools), F5 employees vs subcontractors split, F6 Jobber OAuth pull. Recommends Cloudflare Pages + Workers + D1 + Supabase Auth as system stack (Path 1) over Netlify + Supabase (Path 2). Lists 6 open questions for Cameron (per-account vs per-user, Apple Dev account, GPS coarse vs precise, multi-market calibration, sub pricing inputs, crane sub-tonnage matrix).

## SMS_FALLBACK_SPEC.md

Design draft 2026-05-09 for the SMS-fallback flow. Premise: tree work happens in dead zones where voice/SMS works but data doesn't, so SMS punches through where HTTPS fails. Three architecture options (A activate parked v2.3 Cloudflare Worker — recommended; B standalone tiny worker; C Supabase Edge Functions). Specifies a versioned packet format `TQ#<v>#<acct>#<species>#<dbh>#<height>#<crown>#<trim>#<over48>#<flags>#<zip>` (~46 chars, fits a single 160-char SMS), 3-character species codes (`ACR`/`QRC`/`FRX`/`PIN`/`PCA`/etc.), and Quo (formerly OpenPhone) as the SMS provider rather than Twilio because Cameron is already a Quo user and the unified inbox is a CRM win. Defines the inbound webhook flow, account-to-phone linking via reverse-confirm SMS (`TQLINK 384192`), rate limits (60/hr per linked account, 10/day per anon, 60s dedupe), error states with specific reply text, and a 7-step implementation phasing (SMS-1 activate v2.3 → SMS-7 anon-mode/yard-sign onramp). Open questions: same Quo number or separate, ship anon mode in MVP or defer, reply commands beyond YES/LINK/STOP/HELP, legal disclaimer in every reply, A2P 10DLC inheritance from existing brand reg.

## species_data/EXTRACTION_SUMMARY.md

The data-coverage doc, last updated 2026-05-09. Reports direct API extraction from 14 cities' open-data portals (Socrata/ArcGIS/CKAN) totaling 2.55M trees inventoried (NYC 652k → LA County Parks 4k). Lists confirmed-unavailable cities (Houston, Honolulu, Chicago, Minneapolis). Calls out three cities that expose removal-status data — Seattle (55k removed records, the gold standard with `CURRENT_STATUS='REMOVED'`), Atlanta (16k mortality records), Denver (7k `condition='Dead'`), and notes NYC's removal data is unusable because the species field is null on dead/stump records. Provides the cosmopolitan-species ranking (red maple, honeylocust, Norway maple, Japanese zelkova, ginkgo present in 5–6 cities) which is what TreeQ should always know regardless of region. Documents 10 of 35 TreeQ regions have direct city-derived data; the other 25 are gaps requiring extension publication synthesis (handled qualitatively in `RESIDENTIAL_GAP_AND_SOURCES.md`).

## species_data/MAY_2026_PEST_PRESSURE_MATRIX.md

The "why is this species removal-flagged" doc, current as of May 2026. Tabulates 16 active drivers across the country: EAB (36 states, all Fraxinus YES universally); Beech Leaf Disease (13 states + Ontario); Spotted Lanternfly + Tree of Heaven (OH statewide quarantine Feb 2026); Bradford/Callery Pear bans (OH/SC/PA/NJ; IN moving 2026); Oak Wilt (TX A&M increased cost-share 2026); Sudden Oak Death (16 CA counties + Curry Co OR); HWA (NE/Mid-Atlantic/Central Appalachians); Bronze Birch Borer (high mortality on European white birch); Norway Maple invasive removals (NE/Upper MW/PNW); Silver Maple structural (national); Siberian Elm invasive (Plains/Mountain West/Desert SW); Blue Spruce decline (Mountain West/Plains); Aleppo/Austrian Pine decline; Eucalyptus (CA wildfire/limb failure); Albizia hazard (HI #1 priority); Laurel Wilt (SE redbays/sassafras). Each maps to a YES/MAYBE/NO flag in `removal_priority_top10.csv`. Key insight for Rochester: 8 of the top-10 removal-priority species there are YES-flagged (4 Norway Maple variants + Green Ash + Callery Pear + Silver Maple). Sources cited: USDA APHIS Jan 2026 EAB map, Maine DACF/NYSDEC/Mass.gov, OH ODA Feb 2026, TX A&M, UC IPM, US Forest Service.

## species_data/RESIDENTIAL_GAP_AND_SOURCES.md

The "city-street inventories systematically under-represent yard species" doc. Cites peer-reviewed evidence (ISA Arboriculture & Urban Forestry) that residential forests have more conifers, larger DBH range, more diverse species than street/park inventories. Provides per-region lists of species to *boost* in TreeQ ranking 1–2 ranks above their inventory position because they're under-counted (Northeast: white pine, Norway/blue spruce, hemlock, silver maple, fruit trees, walnut/hickory, Bradford pear; SE: Leyland cypress, loblolly pine, Magnolia grandiflora, slash pine, crape myrtle; Texas: live oak, pecan, bald cypress, Chinese tallow; Midwest: silver maple, cottonwood/willow, walnut/butternut, fruit trees, blue/Norway spruce, boxelder, catalpa/mulberry; Mountain West: blue spruce, aspen, cottonwood; PNW: western red cedar, Douglas-fir, bigleaf maple, fruit; Desert SW/CA: eucalyptus, olive, citrus, pepper, bottlebrush; Hawaii: mango/avocado/lychee/citrus, banyan/monkeypod, albizia). Honest answer on removal-volume datasets: there is no comprehensive public dataset; closest available are TCIA surveys (wrong data), ISA journal i-Tree studies, state Extension after-storm reports, utility vegetation management filings, municipal removal logs (best — Boston BPRD has status field), insurance storm data. Recommends Boston CKAN + NYC `spc_common` filter for status='Removed' as the most-authoritative-available next data step. Concludes with the v1 ranking algorithm: city inventory rank → boost residential-gap species → pin Yes-flagged to top → fall back to regional defaults for gap regions.

## Tl;dr cross-cutting observations

- **The product narrative is coherent.** CONTEXT (math layer) → ROADMAP (product phases) → SMS_FALLBACK_SPEC (concrete first server-trigger) → DEPLOY (parked infra ready to receive) → species_data/* (data substrate). Nothing in the docs contradicts itself; the open questions are explicit.
- **The IP-protection story is the load-bearing architectural decision.** Calibration coefficients + biomass equations + per-class times are what justifies activating `deploy/`. Multiple docs reinforce this — it's also why the SMS spec recommends Option A (activate v2.3) rather than the simpler Option B (tiny standalone worker).
- **The data layer is more advanced than the app layer.** The species CSVs have ~280 unique species across 14 real city APIs and a documented 25-region gap, but the live `index.html` ships with ~56 species. The picker work shipping now is the first time the app surface starts catching up to the data substrate.
- **Quo over Twilio is committed.** This is in the SMS spec and in memory; any architecture R-doc downstream should treat it as a settled decision.
- **Roadmap names Supabase Auth as preferred but ROADMAP.md and SMS_FALLBACK_SPEC.md disagree on whether to also adopt Supabase as the database** (ROADMAP recommends D1; SMS spec mentions Supabase for `sms_quotes` table). R5 needs to resolve this.
- **Rochester is the calibration anchor for everything.** All height/crown coefficients are tuned to Rochester. Multi-market support is open question #4 in ROADMAP — the ranking lists are per-region, the math is single-region.
